# Структура рабочего пространства SiYuan в файловой системе — справочник

> Этот документ описывает, как рабочее пространство SiYuan организовано на диске.
> Он дополняет [`SY-FORMAT.ru.md`](./SY-FORMAT.ru.md): тот документ описывает **внутреннюю** JSON-структуру файла `.sy`, а этот — **общую** структуру рабочего пространства в файловой системе.
> Все выводы проверены на реальном рабочем пространстве и по исходному коду ядра.

## 0. Одним предложением

Рабочее пространство SiYuan — это **самоописывающееся** дерево каталогов: блокноты, документы и ресурсы лежат на диске в виде читаемых файлов и каталогов. **Имя файла — это ID; структура каталогов — это иерархия документов.** Нет никакой бинарной базы данных, фиксирующей, «где находятся документы» — файловая система сама является единственным источником истины. Ядро восстанавливает все блокноты и документы, просто обходя каталоги.

## 1. Дерево-шпаргалка (обзор)

```
<корень рабочего пространства>/   например F:\SiYuan\
├── conf/
│   ├── conf.json              # ★Конфигурация уровня рабочего пространства (внешний вид/система/синхронизация/редактор...)
│   ├── ca.crt / cert.pem / key.pem   # TLS-сертификаты
│   └── windowState.json
├── data/                       # DataDir — корень всех данных блокнотов
│   ├── .siyuan/                # Скрытая конфигурация уровня рабочего пространства (правила *ignore и т. д.)
│   ├── assets/                 # ★Глобальные ресурсы (изображения/аудио/видео/файлы)
│   ├── templates/              # Глобальные шаблоны (.md)
│   ├── widgets/                # Виджеты
│   ├── plugins/                # Плагины
│   ├── emojis/                 # Пользовательские эмодзи
│   ├── snippets/               # Сниппеты кода (CSS/JS)
│   ├── public/                 # Статические ресурсы
│   ├── storage/                # Индексы времени выполнения / кэш / базы данных
│   └── <boxID>/                # ★Блокнот (имя каталога = ID блокнота)
│       ├── .siyuan/
│       │   ├── conf.json       # ★Конфигурация блокнота (BoxConf: имя/значок/закрыт...)
│       │   └── sort.json       # Пользовательская сортировка {docID: порядок}
│       ├── <docID>.sy          # Документ (JSON-дерево; имя файла = rootID документа)
│       └── <docID>/            # ★Дочерние документы этого документа (одноимённая пара)
│           ├── <childID>.sy
│           └── <childID>/ ...  # Вложенность произвольной глубины
├── repo/                       # Репозиторий данных (снимки/синхронизация)
├── history/                    # Архив истории правок
└── temp/                       # Временные файлы / файлы экспорта
```

## 2. Корень рабочего пространства

Корень рабочего пространства выбирается пользователем (например, `F:\SiYuan\`). Является ли каталог валидным рабочим пространством, определяется так: существует `conf/conf.json`, и в нём есть поле `kernelVersion`.

Фиксированные подкаталоги в корне:

| Подкаталог | Назначение |
|---|---|
| `conf/` | Конфигурация уровня рабочего пространства + TLS-сертификаты |
| `data/` | **Все данные блокнотов** (ядро этого документа) |
| `repo/` | Репозиторий снимков данных (используется для синхронизации/истории) |
| `history/` | Архив истории правок |
| `temp/` | Временные файлы / файлы экспорта |
| `corrupted/` | Карантин для повреждённых данных (ядро перемещает их туда автоматически) |

## 3. Верхний уровень `data/`

Первый уровень `data/` смешивает два вида записей:

1. **Зарезервированные каталоги ресурсов:** `.siyuan/`, `assets/`, `templates/`, `widgets/`, `plugins/`, `emojis/`, `snippets/`, `public/`, `storage/`.
2. **Каталоги блокнотов:** каждый — папка, названная по ID блокнота.

### Список зарезервированных имён (★ избегайте при записи файлов)

`IsReservedFilename`:

```go
func IsReservedFilename(baseName string) bool {
    return "assets" == baseName || "templates" == baseName || "widgets" == baseName ||
        "emojis" == baseName || ".siyuan" == baseName || strings.HasPrefix(baseName, ".")
}
```

То есть `assets` / `templates` / `widgets` / `emojis` / `.siyuan`, а также всё, что начинается с `.`, **нельзя** использовать как имя блокнота или документа.

### Как распознаётся блокнот

`ListNotebooks` перечисляет `data/` и для каждого каталога:

1. Пропускает зарезервированные имена;
2. Запись должна быть каталогом (не файлом);
3. Имя каталога должно соответствовать формату NodeID (`ast.IsNodeIDPattern`);
4. Если всё выше выполняется → имя этого каталога и есть `box.ID` (ID блокнота).

## 4. Внутри блокнота (★ главное: парность «родитель — дети»)

Это самое важное соглашение во всей структуре:

> **Представление документа `A` на диске = файл `A.sy` + каталог `A/` (последний существует, только если у A есть дочерние документы). Все дочерние документы A лежат внутри `A/`; если у дочернего есть свои дети, вкладывается каталог `A/B/` — на произвольную глубину.**

Реальный пример (из вывода `find`):

```
20221126104620-m06prws/                            # каталог родительского документа A
20221126104620-m06prws.sy                          # сам родительский документ A
20221126104620-m06prws/20230928134805-z11t56h.sy   # дочерний документ B документа A
20221126104620-m06prws/20230928134805-z11t56h/    # каталог детей B (слой внуков)
20221126104620-m06prws/20230928134805-z11t56h/20240304105333-v7g5j1s.sy
```

Подтверждение в коде:

- Разворачивание детей (`Ls`): для пути `.sy` отбрасывается суффикс `.sy`, проверяется существование одноимённого каталога, и при наличии перечисляются его записи.
- Глубина вложенности (`GetChildDocDepth`): обходит одноимённый каталог тем же способом.

> ⚠️ Это означает: **при перемещении или переименовании документа с детьми нужно также переместить его одноимённый каталог**, иначе дети «осиротеют».

## 5. Метаданные блокнота: `<boxID>/.siyuan/conf.json`

Каждый каталог блокнота содержит скрытую папку `.siyuan/` с файлами:

| Файл | Назначение |
|---|---|
| `conf.json` | Конфигурация блокнота (BoxConf) |
| `sort.json` | Пользовательская сортировка `{docID: порядок}` |

Структура `BoxConf`:

| Поле | Тип | Значение |
|---|---|---|
| `name` | string | Отображаемое имя блокнота |
| `sort` | int | Вес сортировки |
| `icon` | string | Значок (hex-код эмодзи, например `"1f3af"`; либо имя файла пользовательского значка) |
| `closed` | bool | Закрыт ли блокнот |
| `refCreateSaveBox` | string | Целевой блокнот для документов, создаваемых через ссылку на блок |
| `refCreateSavePath` | string | Целевой путь для документов, создаваемых через ссылку на блок |
| `docCreateSaveBox` | string | Целевой блокнот для новых документов |
| `docCreateSavePath` | string | Целевой путь для новых документов |
| `dailyNoteSavePath` | string | Путь сохранения новых ежедневных заметок (поддерживает шаблоны) |
| `dailyNoteTemplatePath` | string | Путь к шаблону новых ежедневных заметок |
| `sortMode` | int | Режим сортировки |

> ⚠️ **ID блокнота НЕ хранится в `conf.json`** — ID является само имя каталога блокнота (см. §3).

Места чтения/записи: `GetConf` / `SaveConf`. Путь жёстко задан как `<DataDir>/<boxID>/.siyuan/conf.json`.

---

## 6. Глобальная конфигурация рабочего пространства: `<workspace>/conf/conf.json`

Важно различать два файла `conf.json`:

| Файл | Расположение | Область действия |
|---|---|---|
| `conf/conf.json` | **Корень рабочего пространства** | Глобальная конфигурация уровня рабочего пространства (внешний вид / языки / система / редактор / синхронизация / репозиторий и т. д.) |
| `<boxID>/.siyuan/conf.json` | **Внутри блокнота** | Только этот блокнот (BoxConf) |

На практике `conf/conf.json` занимает ~42 КБ и хранит внешний вид интерфейса, аккаунт, синхронизацию, ИИ, флеш-карточки и все прочие настройки уровня рабочего пространства.

---

## 7. Скрытая конфигурация уровня рабочего пространства: `data/.siyuan/`

`data/.siyuan/` отличается от `.siyuan/` внутри блокнота — первый хранит правила и состояние **уровня рабочего пространства**:

| Файл | Назначение |
|---|---|
| `syncignore` | Правила игнорирования при синхронизации (в стиле gitignore) |
| `searchignore` | Правила игнорирования при поиске |
| `embeddingignore` | Правила игнорирования при построении ИИ-эмбеддингов |
| `indexignore` | Правила игнорирования при индексации |
| `refsearchignore` | Правила игнорирования при поиске ссылок |
| `publishAccess.json` | Контроль доступа при публикации |
| `filesys_status_check/` | Состояние проверки целостности файловой системы |

> Здесь **нет** `conf.json` — конфигурация рабочего пространства лежит в `conf/conf.json` в корне рабочего пространства (см. §6).

## 8. Ресурсы (assets)

Два сосуществующих расположения:

1. **Глобальный `data/assets/`** — основной. Соглашение об именовании выглядит как `<исходное-имя>-<суффикс-docID>.<расширение>` (например, `640-20240927104411-0jh7x96.webp`). Внутри документов ресурсы указываются относительным путём `assets/xxx`.
2. **Внутри блокнота `<блокнот>/assets/`** — в основном используется встроенным блокнотом руководства пользователя; в обычных пользовательских блокнотах его обычно нет.

Распознавание префикса ссылки на ресурс: допустимыми префиксами ссылок на ресурсы считаются только `assets/`, `emojis/`, `plugins/`, `public/` и `widgets/`.

## 9. Связь ID и путей

### Формат ID

- **ID блокнота:** `YYYYMMDDHHMMSS-xxxxxx` (14-значная метка времени + `-` + 6 случайных символов).
- **ID документа:** `YYYYMMDDHHMMSS-xxxxxxx` (14-значная метка времени + `-` + 7 случайных символов).
- Первые 14 цифр кодируют время создания (извлекается функцией `TimeFromID`).
- Алфавит случайной части — `[a-z0-9]`.

### Имя файла = ID (жёсткое ограничение)

- Имя `.sy`-файла документа (без `.sy`) = rootID документа.
- Имя каталога блокнота = ID блокнота.
- Ядро проверяет соответствие `ID имени файла == root.ID` и автоматически исправляет расхождения.

### Абсолютный путь документа

Абсолютный путь документа на диске = `data/<boxID>/<относительный-путь>`, где `<относительный-путь>` выглядит как `/20221126104620-m06prws/20230928134805-z11t56h.sy` (POSIX-стиль, ведущий `/`).

Извлечение ID документа из пути (`GetTreeID`): просто `filepath.Base(path)` с удалённым суффиксом `.sy`.

## 10. Как вычисляется HPath (человекочитаемый путь)

HPath **не хранится** — он вычисляется на лету при загрузке документа (`LoadTreeByData`):

1. Относительный путь документа разбивается по `/`, отбрасываются ведущий пустой сегмент и завершающий сегмент самого документа — остаются **сегменты ID каждого документа-предка**.
2. Для каждого ID предка строится `<ancestorID>.sy` и читается его `title` через `DocIAL()` (**потоковое чтение только `Properties`**, без разбора всего дерева).
3. Заголовки соединяются через `/`, добавляется заголовок текущего документа → получается HPath вида `/Родитель/Ребёнок/Текущий`.
4. Если какой-то `.sy` предка отсутствует, ядро **автоматически создаёт** родителя `Untitled` (issue #7376).

> Это фундаментальная причина, почему «структура каталогов — это данные»: **перемещение или переименование файла/каталога напрямую меняет HPath документа и хлебные крошки** — потому что HPath целиком выводится из цепочки каталогов.

## 11. Замечания о прямой работе с файловой системой

> В соответствии с [SY-FORMAT.ru.md §0.5](./SY-FORMAT.ru.md): **предпочитайте HTTP API / MCP / CLI** для изменения данных — тогда ядро само позаботится о синхронизации индексов. Прямые операции с файловой системой оправданы только для массовой офлайн-миграции, холодной инициализации и подобных сценариев.

При прямой записи файлов:

1. ☐ Имена блокнотов/документов должны соответствовать формату NodeID и **избегать зарезервированных имён** (§3).
2. ☐ Имя `.sy`-файла документа равно его rootID (§9).
3. ☐ У документа с детьми должен быть **парный одноимённый каталог**; перемещение родителя означает и перемещение этого каталога (§4).
4. ☐ Имя/значок/состояние «закрыт» блокнота живут в `<boxID>/.siyuan/conf.json`, а не в `.sy` (§5).
5. ☐ После изменений обычно нужно **перестроить индекс**, иначе поиск/ссылки на блоки/хлебные крошки устареют.
